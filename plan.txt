## Day 1 — Requirements + Scope + What “Production Ready” Means

### Learn

* What a **Landing Zone** is (CAF conceptually)
* Why enterprises split: **Platform vs Workloads**
* What “production-ready” actually includes: governance, security, monitoring, cost, automation

### Do

* Write a 1-page “Client Requirements” doc:

  * Regions: **Canada Central** primary
  * Environments: **Dev + Prod**
  * Network: hub-spoke
  * No public access to PaaS storage
  * Central logging and security monitoring
  * Separate identity duties vs infra duties
* Define non-functional requirements:

  * Availability goals, auditability, cost controls, least privilege

### Output

* Requirements doc (markdown)
* List of “must have / nice to have”

### You must be able to explain

* Why landing zones exist (control + scale + repeatability)

---

## Day 2 — Architecture v1 (High-Level Diagram)

### Learn

* Management Groups vs Subscriptions vs Resource Groups
* Enterprise-scale governance hierarchy

### Do

* Design hierarchy:

  * Root MG
  * Platform MG
  * Prod MG
  * Dev MG
* Plan subscriptions:

  * Platform subscription (hub/shared)
  * Dev subscription
  * Prod subscription
* Create your **High-Level Architecture Diagram**

  * Governance layer
  * Network layer
  * Security layer
  * Observability layer

### Output

* Diagram v1 (clean, presentable)

### You must be able to explain

* Why subscriptions are split and what goes into Platform vs Workloads

---

## Day 3 — Identity & RBAC (Production Access Model)

### Learn

* Microsoft Entra ID basics for cloud governance
* RBAC vs Azure Policy (who can vs what is allowed)
* Privileged access patterns: break-glass, least privilege, separation of duties

### Do

* Define personas + roles:

  * Platform Admin (limited)
  * Network Admin
  * Security Admin
  * App Team Dev
  * App Team Prod
  * Reader/Auditor
* Map each persona to:

  * Management Group scope / Subscription scope
  * Built-in roles (Owner/Contributor/Reader + specialized roles)
* Design **break-glass** plan (2 accounts, emergency only)

### Output

* RBAC matrix (table)
* Access model document

### You must be able to explain

* How you prevent “everyone is Owner” chaos

---

## Day 4 — Azure Policy Strategy (Guardrails)

### Learn

* Policy vs Initiative
* Effects: Deny, Audit, DeployIfNotExists, Modify
* Policy assignment scope inheritance

### Do

Design **production policy baseline**, grouped as initiatives:

* **Tagging** required: Environment, Owner, CostCenter, DataClassification
* Restrict allowed regions (Canada only)
* Deny public IP creation (or audit + exception process)
* Enforce diagnostic settings (DeployIfNotExists concept)
* Allowed VM SKUs (cost control)
* Require secure transfer for Storage
* Audit NSG rules that allow 0.0.0.0/0 inbound

### Output

* Policy baseline document:

  * Policy list + purpose + scope + exceptions plan

### You must be able to explain

* How policy reduces risk *without blocking teams permanently*

---

## Day 5 — Network Design (IP Plan + Subnets)

### Learn

* CIDR planning
* Hub-spoke patterns
* DNS needs with Private Link
* Route tables and why they matter

### Do

* Create address plan:

  * Hub VNet CIDR
  * Dev Spoke CIDR
  * Prod Spoke CIDR
* Define subnet layout:

  * Azure Firewall subnet (required name conventions)
  * Bastion subnet (optional but good)
  * Management subnet
  * Workload subnets (Dev/Prod)
  * Private endpoint subnet (or per-service)
* Define peering rules:

  * Use hub as central routing and services

### Output

* Network design document + subnet table

### You must be able to explain

* Why you chose those CIDRs and how you prevent IP conflicts later

---

## Day 6 — Hub-Spoke Implementation (Connectivity)

### Learn

* VNet peering behavior and limitations
* Hub services concept: firewall, bastion, VPN/ER gateways
* Spoke isolation principles

### Do

* Implement the network:

  * Hub VNet
  * Dev Spoke
  * Prod Spoke
  * Peering with correct flags (gateway transit plan conceptually)
* Validate connectivity goals:

  * Spokes can reach shared services in hub
  * Spokes remain isolated unless explicitly allowed

### Output

* Updated diagram showing peering + flows

### You must be able to explain

* What “transitive routing” is and why peering alone doesn’t equal full routing

---

## Day 7 — Routing & Azure Firewall (Traffic Control)

### Learn

* UDR (user-defined routes) in Azure
* Forced tunneling / default routes
* Firewall rule types:

  * Network rules
  * Application rules
  * DNAT (concept)
* Egress control patterns (most important in enterprises)

### Do

* Design routing:

  * Spokes route 0.0.0.0/0 through firewall (where appropriate)
  * Exceptions: Azure service endpoints/private link
* Design firewall policy:

  * Minimal outbound allowlist approach (documented)
  * Logging enabled
* Threat model:

  * What happens if a VM is compromised?

### Output

* Routing table design + firewall policy document

### You must be able to explain

* How you stop “anything can go anywhere” outbound traffic

---

## Day 8 — Private Endpoints + Private DNS (No Public PaaS)

### Learn

* Private Link vs Service Endpoints (differences + when to use)
* Private DNS zone linkage
* Name resolution flow for private endpoints

### Do

* Choose 2–3 services to secure with Private Endpoint:

  * Storage account (must)
  * Key Vault (strongly recommended)
  * SQL (optional)
* Design:

  * Where private endpoints live (which subnet)
  * Private DNS zones required and how they link to VNets
* Validate “public access disabled” design

### Output

* Private Link design doc + DNS flow diagram

### You must be able to explain

* Why private endpoints matter, and what breaks if DNS is wrong

---

## Day 9 — Secrets & Key Vault + Managed Identity

### Learn

* Why “secrets in code/pipelines” is a disaster
* Managed Identity basics and patterns
* Key Vault access control options (RBAC vs access policies conceptually)

### Do

* Design:

  * Key Vault placement (subscription/resource group)
  * Access rules (who/what can read secrets)
  * Rotation strategy (even if manual)
* Create a scenario:

  * App reads secret via Managed Identity
  * Ops can rotate secret without code changes

### Output

* Identity-to-KeyVault access flow diagram + policy doc

### You must be able to explain

* How Managed Identity reduces secret leakage risk

## Day 10 — Logging, Monitoring, and Diagnostics (Production Observability)

* Azure Monitor, Log Analytics, Activity Logs

    * Firewall
    * Key services (Storage, Key Vault)
    * Activity logs per subscription
* Alert plan:

  * Security: policy changes, firewall denies spikes
  * Infra: CPU, disk, availability
  * Cost: budget thresholds

### Output

* Monitoring plan + alert matrix

### You must be able to explain

* How you detect incidents and prove auditability
